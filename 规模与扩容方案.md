# 规模与扩容方案（500 人 → 1000 人 → 10000 人）

## 一、规模概览

| 阶段     | 用户量   | 推荐方案           | 说明 |
|----------|----------|--------------------|------|
| **初期** | 约 500 人 | Supabase Free + 静态站点 | 验证业务，成本为 0，可随时升级 |
| **成长** | 约 1000 人 | 同上或 Supabase Pro | 仍在 Free 限额内可继续用；接近限额再升 Pro |
| **扩展** | 约 10000 人 | Supabase Pro（或 Team）+ 同一套前端 | 按用量付费，预留存储与请求余量 |

---

## 二、容量粗算（供参考）

- **云端**  
  - 每用户一行 `users` 表，`data` 为 JSON（班级、学生、积分等）。  
  - 粗估：约 **100KB～500KB/用户**（含班级、学生、勋章等），照片存 URL 或外链，不放大 JSON。  
  - 500 人 ≈ 50～250MB；1000 人 ≈ 100～500MB；10000 人 ≈ 1～5GB。

- **请求量**  
  - 实时同步：有改动才上传，防抖 2.5 秒 + 2 分钟/5 次才发一次，平均每用户每天几十～几百次 API 足够。  
  - 500～1000 人：Supabase Free 的「Unlimited API requests」足够（注意合理限流即可）。  
  - 10000 人：Pro 档仍以请求次数为主，一般够用；若遇限流再配合前端节流/队列。

- **前端**  
  - 纯静态（HTML/JS/CSS），可放任意静态托管 + CDN，**不按用户数扩容**，500 与 10000 人同一套前端即可。

---

## 三、按阶段的推荐配置

### 1. 初期约 500 人

- **Supabase**  
  - 使用 **Free 计划**。  
  - 在 SQL Editor 执行 `supabase-schema.sql` 建表。  
  - 同一项目可同时支撑 500～1000 人（在 50k MAU、500MB 数据库内）。

- **前端**  
  - 部署到 **GitHub Pages / Vercel / Netlify** 等任一静态托管即可。  
  - 所有用户共用一个 Supabase 项目（同一 `supabaseUrl` + `supabaseKey`）。

- **注意**  
  - Free 项目约 1 周无活跃会暂停，有访问即恢复；若需 7×24 不暂停，可提前升 Pro。

### 2. 约 1000 人

- **继续用 Free**  
  - 若数据库体积仍在 500MB 内、MAU 在 5 万内，无需改架构，只做监控（见下文）。

- **准备升级时**  
  - 在 Supabase Dashboard 看 **Database → Size**、**Settings → Usage（MAU）**。  
  - 接近 500MB 或 5 万 MAU 时，升级到 **Pro（$25/月起）**，获得更大数据库与 MAU 额度。

- **前端**  
  - 无需改动，仍是同一套静态站 + 同一 Supabase 项目。

### 3. 约 10000 人（可升级到 1 万人）

- **Supabase**  
  - 使用 **Pro 计划**（约 $25/月起 + 超出部分用量计费）。  
  - Pro 一般包含：更大数据库（如 8GB 起）、更高 MAU（如 10 万）、备份与更好 SLA，足以支撑 1 万人。

- **数据库**  
  - 保持当前表结构（`users` 单表、jsonb `data`）即可。  
  - 若单行 `data` 很大（如单用户 >1MB），可考虑：  
    - 把照片/大对象存 Supabase Storage，`data` 里只存 URL；  
    - 或按需做「冷数据」归档（历史记录单独表），控制单行体积。

- **连接与限流**  
  - 前端直连 Supabase（无自建后端），Supabase 会管理连接池。  
  - 已在用的「防抖 + 离线队列 + 2 分钟/5 次再同步」能显著降低并发请求，一般不需要再改。  
  - 若 Supabase 侧出现限流，可在前端略加强节流（例如同一用户最少间隔 1 分钟再发一次上传）。

- **前端与 CDN**  
  - 仍为静态资源，放同一静态托管即可；用 Vercel/Netlify 等自带 CDN，即可支撑 10000 人访问。

- **后续若再扩大**  
  - 可考虑 **Supabase Team** 或更大计算/存储套餐；  
  - 或把「读多写少」的统计类需求迁到只读副本（若 Supabase 提供），当前业务以「每用户写自己一行」为主，单项目 Pro 足够到 1 万人。

## 四、部署清单（便于实施与升级）

- [ ] **Supabase**  
  - 建项目 → 执行 `supabase-schema.sql` → 记下 **Project URL** 和 **anon key**。

- [ ] **前端**  
  - 在 `app.js` 中配置 `supabaseUrl`、`supabaseKey`（或通过环境变量注入）。  
  - 构建/打包后部署到静态托管（如 `dist` 或当前目录上传）。

- [ ] **域名（可选）**  
  - 为前端绑定自有域名，便于长期使用与品牌统一。

- [ ] **监控（建议 1000 人起做）**  
  - 在 Supabase Dashboard 定期看：Database size、API 请求量、MAU。  
  - 设置用量告警（若 Supabase 支持），接近 Free 限额时提前规划升级 Pro。

- [ ] **升级到 10000 人时**  
  - 升级到 Pro → 再次核对 Database size 与 MAU 是否在套餐内。  
  - 前端无需换架构，同一套代码、同一项目即可支撑到 1 万人；若单用户数据量很大再按上文优化 `data` 体积。

---

## 五、简要结论

- **500 人**：Supabase Free + 静态前端即可，成本为 0，部署简单。  
- **1000 人**：多数情况仍 Free；接近限额再升 Pro，前端不变。  
- **10000 人**：Supabase Pro + 同一套前端即可支撑；注意控制单用户 `data` 体积与请求频率，后续可按需再升级 Team 或更大套餐。

整体上**无需为 500/1000/10000 设计不同架构**，只需按用量选 Supabase 档位并做好监控与升级节奏即可。
