# 实时云存储方案与部署指南

## 一、目标与原则

- **不丢数据**：本地优先写入，再异步上报云端；失败时进入离线队列，有网后自动重试。
- **尽量实时**：有变更后短时间防抖即上报（如 2～3 秒），减少“刚改完就关页面”导致未同步的风险。
- **可离线使用**：无网时照常读写本地，恢复网络后自动同步到云并拉取他人/他端更新（若有多端）。

---

## 二、整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│  用户操作（保存、加减分、新增学生等）                                  │
└─────────────────────────────┬─────────────────────────────────────┘
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  1. 立即写本地（localStorage / IndexedDB）                          │
│     → 保证关页面、断网也不丢数据                                     │
└─────────────────────────────┬─────────────────────────────────────┘
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  2. 标记 dataChanged，触发 scheduleSync（防抖 2～3 秒）            │
└─────────────────────────────┬─────────────────────────────────────┘
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  3. 有网：调用 syncToCloud() 上传到 Supabase                       │
│     无网/失败：将本次 payload 入「离线队列」                         │
└─────────────────────────────┬─────────────────────────────────────┘
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  4. 网络恢复 / 页面重新可见：先消费离线队列重试，再按需拉取云端最新    │
└─────────────────────────────────────────────────────────────────┘
```

- **实时**：通过“短防抖 + 有变更就尝试上传”实现，在保证不刷爆接口的前提下尽量快。
- **不丢**：本地先写 + 失败入队 + 上线重试，三重保障。

---

## 三、同步策略（前端逻辑）

| 场景           | 行为 |
|----------------|------|
| 每次保存/变更  | 先写本地，再 scheduleSync（防抖 2～3 秒） |
| 防抖到时       | 若有网且 dataChanged，执行 syncToCloud |
| syncToCloud 失败 | 把当前 payload 写入离线队列（按用户维度） |
| 上线 / visibility 可见 | 先按队列顺序重试上传，再可选执行 syncFromCloud |
| 定时           | 每 N 分钟若 dataChanged 再尝试一次云端同步（兜底） |

冲突与一致性：

- 以 **lastModified / updated_at** 为准，**新覆盖旧**（当前已实现）。
- 云端只存“最新一份”全量数据，不做多版本合并，逻辑简单、易部署。

## 四、云端：Supabase 表与 RLS

### 4.1 表结构

在 Supabase SQL Editor 中执行（或直接使用项目中的 `supabase-schema.sql` 文件）：

```sql
-- 用户数据表（与当前前端一致）
create table if not exists public.users (
  id text primary key,
  data jsonb not null default '{}',
  updated_at timestamptz not null default now()
);

-- 可选：便于排查与统计
comment on table public.users is '童心宠伴用户数据，data 为全量 JSON';
create index if not exists users_updated_at_idx on public.users (updated_at desc);

-- 开启 RLS（按需启用）
alter table public.users enable row level security;

-- 策略：允许匿名/已认证用户通过 service key 或 anon key 读写（与当前前端一致）
-- 若仅前端 anon key 访问，可允许所有人对自己的 id 行 upsert/select
create policy "users_upsert_select"
  on public.users
  for all
  using (true)
  with check (true);
```

若你希望**按用户隔离**（每人只能读写自己的行），可改为：

```sql
create policy "users_own_row"
  on public.users
  for all
  using (auth.uid()::text = id)
  with check (auth.uid()::text = id);
```

（当前前端是 anon 直连、用 `id` 为业务用户 ID，若未用 Supabase Auth，上面 `auth.uid()` 需按你实际认证方式改。）

### 4.2 部署步骤（Supabase）

1. **创建项目**  
   登录 [Supabase](https://supabase.com) → New Project → 选区、设库密码。

2. **建表**  
   在 SQL Editor 中执行上面 `create table` 和索引、注释部分。

3. **拿到 URL 和 Key**  
   Settings → API：  
   - `Project URL` → 前端 `supabaseUrl`  
   - `anon` `public` key → 前端 `supabaseKey`（仅前端用 anon 即可）

4. **配置前端**  
   在 `app.js` 中（或通过构建时环境变量）设置：

   - `supabaseUrl`: 你的 Project URL  
   - `supabaseKey`: anon public key  

   若用环境变量，可建 `.env`（不要提交密钥）：

   ```env
   VITE_SUPABASE_URL=https://xxx.supabase.co
   VITE_SUPABASE_ANON_KEY=eyJ...
   ```

5. **（可选）Realtime**  
   若以后要做多端/多人实时看到同一数据变化，可在 Supabase 中对该表开启 Realtime，前端订阅 `users` 表变更；当前“短防抖 + 离线队列”已能保证数据不丢与准实时。

## 五、前端实现要点（与本仓库对应）

- **离线队列**  
  - 键：`class_pet_sync_queue`（或按用户分 key）。  
  - 值：队列数组，每项为 `{ payload, ts }`，payload 即当前要上传的 `{ id, data, updated_at }`。  
  - syncToCloud 失败（网络/5xx）时 push 入队；上线/visibility 时消费队列并重试。

- **实时上报**  
  - scheduleSync 防抖改为 2～3 秒。  
  - syncData 条件放宽：例如“距上次同步 > 2 分钟或变更次数 ≥ 5”即上传，避免 15 分钟太长。

- **重试与退避**  
  - 队列重试：每次取队首上传，成功则删队首并继续，失败则保留并下一轮再试。  
  - 可选：连续失败 3 次后延迟 30 秒再试，避免死循环刷接口。

---

## 六、如何验证“数据不丢”

1. **有网**：改几条数据 → 等几秒 → 到 Supabase Table Editor 看 `users` 表对应 `id` 的 `data`、`updated_at` 是否更新。  
2. **无网**：断网 → 改数据 → 关标签页 → 再开、连网 → 看是否自动同步到云端（队列重试 + 正常 syncToCloud）。  
3. **多端**：A 端改数据并同步成功，B 端刷新或重新进入后拉取，应以最新 `updated_at` 为准得到 A 的修改。

---

## 七、小结

| 项目     | 说明 |
|----------|------|
| 实时     | 短防抖（2～3 秒）+ 有变更即尝试上传 + 可选定时兜底 |
| 不丢     | 本地先写 + 失败入离线队列 + 上线/可见时重试 |
| 部署     | Supabase 建表 + 配 URL/Key + 前端配置同上 |
| 冲突     | 以时间为准，新覆盖旧，单表全量 JSON 存储 |

按此方案部署并接好前端后，即可在“实时云存储”下尽量保证数据不丢失；后续若需多端实时刷新，再叠加 Supabase Realtime 即可。
